## Overview

Onlooker Desktop ships as a single app that operates in two fundamentally different modes.
Understanding the distinction is essential before reading the rest of this spec.

### The two modes are different products

| | Self-hosted | Managed |
|---|---|---|
| **Core job** | Observability companion for Claude Code | Standalone Claude interface with observability built in |
| **Who it's for** | Developers already using Claude Code | Anyone who wants Claude + observability without Claude Code setup |
| **Has chat?** | No | Yes — it is the chat interface |
| **Has observability?** | Yes — reads from `~/.claude/onlooker/logs/` | Yes — plugins run inside the app |
| **API key** | Not required (optional for Counsel synthesis) | Onlooker's key, proxied |
| **Claude Code required** | Yes | No |
| **Price** | Free forever | Subscription (Free / Pro / Power) |

The self-hosted mode is documented in full in `SELF_HOSTED_SPEC.md`. This document covers
managed mode only.

### The managed mode value proposition

Managed mode is for the person who:
- Wants to use Claude in a purpose-built desktop client, not a browser tab
- Wants observability (quality scores, event feed, weekly review) without setting up Claude Code plugins
- Doesn't want to manage an Anthropic API key or billing account
- Is willing to pay a flat monthly fee for a zero-friction experience

The chat interface is not an afterthought — it *is* the product for managed users. The
observability layer wraps it, making every session inspectable in ways the standard
Claude interfaces don't provide.

## 1. What Managed Mode Looks Like

The managed app UI is the original split-pane layout from the pre-Phase-1 codebase:
**chat pane on the left, observability panel on the right** — plus the four-view sidebar
for Sessions, Metrics, and Weekly Review.

```
┌──────┬────────────────────────────┬─────────────────────────┐
│      │                            │                         │
│  ✦   │   Chat                     │   Onlooker              │
│      │   ─────────────────────    │   ─────────────────     │
│  ⚡  │   Messages thread          │   Live Feed   Metrics   │
│      │                            │                         │
│  ◎   │   [streaming response]     │   ⚡ SEN  PreToolUse    │
│      │                            │   ✓ TRB  Score 0.87    │
│  ▦   │                            │   ℹ ARC  Memory write  │
│      │   ────────────────────     │                         │
│  ☆   │   [input]          [↑]     │   Debug                 │
│      │                            │                         │
│  ⚙   │                            │                         │
└──────┴────────────────────────────┴─────────────────────────┘
```

The observability panel shows events generated by the **bundled plugins** as they respond
to conversation events — not real Claude Code hook events, but a close approximation:

- **Sentinel** — analyses Claude's response for described destructive actions; flags them
- **Tribunal** — scores each response for quality, confidence, and bias
- **Archivist** — writes session summaries at turn boundaries
- **Scribe** — captures user intent at the start of each turn

These events write to `~/.claude/onlooker/logs/` exactly as they would in self-hosted mode, so
the Sessions, Metrics, and Weekly Review views work identically across both modes.

## 2. Architecture

### 2.1 System topology

```
┌──────────────────────────────────────────────────────────┐
│                  Onlooker Desktop (managed)              │
│                                                          │
│  ┌─────────────────────┐   ┌────────────────────────┐   │
│  │   Chat pane          │   │  Observability panel   │   │
│  │   (renderer)         │   │  (renderer)            │   │
│  └──────────┬───────────┘   └───────────┬────────────┘   │
│             │                           │                 │
│  ┌──────────▼───────────────────────────▼────────────┐   │
│  │              Main process                         │   │
│  │  mode: "managed"                                  │   │
│  │  auth: JWT stored in OS keychain                  │   │
│  │  plugins: bundled in resources/plugins/           │   │
│  └──────────────────────────┬────────────────────────┘   │
└─────────────────────────────┼──────────────────────────  ┘
                              │ HTTPS
                              ▼
           ┌──────────────────────────────────┐
           │      api.onlooker.dev            │
           │                                  │
           │  • JWT verification              │
           │  • Per-user quota enforcement    │
           │  • Token metering                │
           │  • Message-blind forwarding      │
           │  • Abuse detection               │
           └─────────────────┬────────────────┘
                             │
                             ▼
                  Anthropic API
```

### 2.2 Why a proxy, not a bundled key

A key embedded in the Electron binary can be extracted with `strings` or by unpacking the
`.asar` archive. For a subscription product this creates two unsolvable problems:

1. **No enforcement** — usage limits cannot be enforced client-side
2. **No revocation** — a leaked key requires shipping a new binary to rotate it

The proxy is intentionally thin and stateless. It forwards requests, enforces quotas, and
records only metadata (user ID, timestamp, token counts, model). **No message content is
ever stored server-side.** All JSONL logs remain local to the user's machine.

### 2.3 Proxy server contract

```
POST /v1/chat
  Authorization: Bearer <onlooker_jwt>
  Body: { messages, model, max_tokens, session_id }

  1. Verify JWT (signature + expiry)
  2. Look up user quota in Redis
  3. If over limit → 429 with { quota, used, reset_date }
  4. Forward to Anthropic API (streaming)
  5. Stream response tokens back to client
  6. On stream end → increment usage counter in Redis
```

The proxy does not inspect or log message content at any point. This is both a privacy
commitment and a simplicity choice — keeping the proxy message-blind means it has no
GDPR surface area around conversation data.

### 2.4 Authentication flow

```
1. User selects "Managed" on first launch
2. App opens onlooker.dev/activate in default browser
3. User signs up / logs in, selects tier, completes payment
4. onlooker.dev redirects to: onlooker://auth?token=<refresh_token>
5. App receives deep link → stores refresh token in OS keychain
6. App exchanges refresh token for short-lived JWT (24h)
7. All proxy calls carry JWT in Authorization header
8. App silently refreshes JWT before expiry
```

The browser handles all account creation and payment — no sensitive flows inside Electron.
Deep link registration in `package.json`:
```json
"protocols": [{ "name": "Onlooker", "schemes": ["onlooker"] }]
```

## 3. Pricing Tiers

### 3.1 Tier table

| Tier | Price | Messages/mo | Models | Plugins | Support |
|------|-------|-------------|--------|---------|---------|
| **Free** | $0 | 50 | Haiku 4.5 | Sentinel, Scribe | Community |
| **Pro** | $18/mo | 1,000 | Sonnet 4.6, Haiku 4.5 | All (see §4) | Email |
| **Power** | $42/mo | 4,000 | Sonnet 4.6, Haiku 4.5 | All + Echo | Priority |

Self-hosted mode is always free and unlimited — those users pay Anthropic directly.

### 3.2 Pricing rationale

Current Anthropic API costs for Sonnet 4.6:
- Input: $3 / 1M tokens · Output: $15 / 1M tokens

Average managed conversation turn: ~2K input + ~800 output tokens
- Cost per turn ≈ $0.006 + $0.012 = **~$0.018**

| Tier | Messages | COGS at list price | Price | Gross margin |
|------|----------|-------------------|-------|--------------|
| Free | 50 | ~$0.90 | $0 | Acquisition cost |
| Pro | 1,000 | ~$18 | $18 | ~0% at list, improves with volume |
| Power | 4,000 | ~$72 | $42 | Negative at list |

This looks alarming for Power tier but the math improves significantly with Anthropic
volume discounts. At meaningful scale (>$10K/mo spend), negotiated rates typically run
40–60% below list. Target gross margin of 40–50% at steady state.

**Launch recommendation**: Start with Pro at $18/mo. Power tier pricing should not be
finalised until you have enough users to negotiate committed-use pricing with Anthropic.
Consider Power as an "invite-only beta" at launch to control COGS exposure.

### 3.3 Limit enforcement

Limits are enforced server-side at the proxy. The client shows a live meter in the
sidebar footer:

```
Pro  ·  247 / 1,000  ·  resets May 1
████████░░░░░░░░░░░░░  24.7%
```

At **80%**: non-blocking banner in the chat pane — "You've used 800 of 1,000 messages
this month. [Upgrade →]"

At **100%**: inline message in the chat thread (not a modal):
```
  You've reached your Pro plan limit for this month.
  Upgrade to Power for 4,000 messages, or wait until May 1.
  [Upgrade to Power →]   [Wait until reset]
```

## 4. Plugin Bundling

### 4.1 What bundled means

In self-hosted mode, plugins are shell scripts Claude Code invokes via hooks. In managed
mode, Claude Code is not installed — the Electron main process runs the plugin lifecycle
directly, triggered by conversation events rather than tool call hooks.

Bundled plugin scripts live in `resources/plugins/` inside the packaged app. The
plugin bridge resolves them there when `mode === "managed"`:

```javascript
function resolveScript(plugin, command) {
  if (store.get("mode") === "managed") {
    return path.join(process.resourcesPath, "plugins", plugin, "scripts", command + ".sh");
  }
  return path.join(os.homedir(), ".claude", "plugins", plugin, "scripts", command + ".sh");
}
```

### 4.2 Plugin inventory by tier

| Plugin | Free | Pro | Power | Trigger in managed mode |
|--------|------|-----|-------|------------------------|
| **Sentinel** | ✓ | ✓ | ✓ | Runs after each assistant response; parses for described destructive actions |
| **Scribe** | ✓ | ✓ | ✓ | Runs on each user message; captures intent |
| **Tribunal** | — | ✓ | ✓ | Runs after each assistant response; scores quality |
| **Archivist** | — | ✓ | ✓ | Runs at session end; writes memory summary |
| **Counsel** | — | ✓ | ✓ | Runs weekly on demand; generates Weekly Review synthesis |
| **Echo** | — | — | ✓ | Runs on demand from Weekly Review; prompt regression CI |
| **Forge** | — | — | — | Not bundled — developer scaffolding tool, not relevant here |

### 4.3 The capability gap vs self-hosted

This is the most important thing to communicate clearly in marketing:

**Self-hosted** plugins hook into real Claude Code tool events — every bash command, file
write, and web fetch is gated by Sentinel and scored by Tribunal at the moment it happens.

**Managed** plugins operate on conversation text only. Sentinel parses Claude's *descriptions*
of actions rather than observing the actions themselves. Tribunal scores response quality
but cannot see tool outputs.

This is not a failure of managed mode — it's a different use case. Managed mode is for
conversational AI use, not agentic coding. The observability layer still provides genuine
value: quality scoring, intent tracking, memory, and weekly review all work meaningfully
on conversation data.

The UI is explicit about this distinction. The observability panel footer reads:
- Self-hosted: `claude-sonnet-4 · 4 hooks active · ~/.claude/onlooker/logs`
- Managed: `claude-sonnet-4 · managed · conversation events`

### 4.4 Plugin versioning and updates

Bundled plugins are packaged as `plugins.asar` inside the app bundle, versioned
independently of the main app. Plugin updates can ship as delta downloads without a
full app release.

The app checks `onlooker.dev/api/plugin-versions` on launch and prompts for updates
non-disruptively: a dot on the Settings icon, with details inside Settings → Plugins.

### 4.5 JSONL compatibility

Bundled plugins write the same JSONL event format as self-hosted plugins. The log
watcher, Live Feed, Sessions, Metrics, and Weekly Review views are identical in both
modes. This is a hard requirement — the JSONL format is the contract.

## 5. Onboarding Flow

### 5.1 First-launch mode selection

On first launch, if no auth token and no `~/.claude/onlooker/logs/` with existing sessions
are found, the app shows a mode selection screen:

```
┌──────────────────────────────────────────────────────────┐
│                     ✦  Onlooker                          │
│          AI agent observability for Claude               │
│                                                          │
│  ┌───────────────────────┐  ┌───────────────────────┐   │
│  │  ✦  Managed           │  │  ⌘  Self-hosted        │   │
│  │                       │  │                        │   │
│  │  Zero setup.          │  │  For Claude Code       │   │
│  │  Chat with Claude     │  │  users.                │   │
│  │  and observe your     │  │                        │   │
│  │  sessions — all in    │  │  Observe your Claude   │   │
│  │  one place.           │  │  Code sessions.        │   │
│  │                       │  │  No API key needed.    │   │
│  │  Free · Pro · Power   │  │  Free forever.         │   │
│  │                       │  │                        │   │
│  │  [Get started →]      │  │  [Set up →]            │   │
│  └───────────────────────┘  └───────────────────────┘   │
└──────────────────────────────────────────────────────────┘
```

This screen is shown once. The mode choice is stored in settings and can be changed
in Settings → Account.

### 5.2 Managed onboarding

```
Step 1  Mode selection → "Get started"
Step 2  Tier selection (Free / Pro / Power cards with feature comparison)
Step 3  "Create account" → opens onlooker.dev/signup in default browser
Step 4  User signs up + optionally subscribes (Stripe, on the website)
Step 5  Website redirects to onlooker://auth?token=<refresh_token>
Step 6  App receives deep link, stores token in keychain, exchanges for JWT
Step 7  Success screen: "You're connected. Start chatting."
Step 8  App opens to Chat view
```

### 5.3 Self-hosted onboarding

Self-hosted onboarding is handled entirely by the existing `Onboarding.jsx` component
(four states: fresh / no_plugins / no_logs / done). No changes needed — it does not
ask for an API key or account.

### 5.4 Mode switching

Users can switch from managed to self-hosted (or vice versa) in Settings → Account.
Switching modes does not delete local JSONL logs — session history is preserved.

## 6. Managed-Mode UI Differences

The managed app includes everything in the self-hosted spec (Live Feed, Sessions,
Metrics, Weekly Review) plus:

| Addition | Details |
|----------|---------|
| **Chat pane** | Full-width left panel with streaming, message history, debug overlay |
| **Usage meter** | Sidebar footer + Settings panel showing messages used / quota / reset date |
| **Upgrade prompts** | Inline in chat at 80% and 100% usage; never modal |
| **Account panel** | Settings → Account shows tier, usage history, billing link (opens browser) |
| **Model selector** | Settings → Model shows only models available on current tier |
| **Mode badge** | Sidebar footer reads `managed` instead of `4 hooks active` |

The observability panel in managed mode has the same three tabs (Live Feed / Metrics /
Debug) as the self-hosted right panel from the original codebase — it sits alongside
the chat rather than being the primary view.

## 7. Implementation Phases

### Phase 1 — Self-hosted MVP (current focus)
Covered in `SELF_HOSTED_SPEC.md`. No managed-specific work yet.

### Phase 2 — Managed auth + proxy
- `onlooker://` deep link handler in `main/index.js`
- JWT auth flow, keychain storage, silent refresh
- Proxy server (`api.onlooker.dev`) — Fastify, Redis quota, Fly.io deploy
- Mode selection screen on first launch
- Usage meter in sidebar footer + Settings
- Re-introduce chat pane (from pre-Phase-1 codebase) gated behind managed mode

### Phase 3 — Plugin bundle
- Package Sentinel, Scribe, Tribunal, Archivist, Counsel into `resources/plugins/`
- `plugins.asar` build step in electron-builder
- Managed-mode plugin bridge path resolution
- Conversation-event triggers for each plugin
- Plugin update mechanism

### Phase 4 — Billing + polish
- Stripe on onlooker.dev (not in the app)
- Webhook → quota update in Redis
- Tier-aware model selector
- Upgrade prompts at 80% / 100%
- Usage history view on onlooker.dev

| # | Question | Notes |
|---|----------|-------|
| 1 | Free tier: Haiku only or rate-limited Sonnet? | Haiku is sustainable at $0; Sonnet on free risks abuse but converts better |
| 2 | Power tier pricing: hold at $42 or wait for volume discounts? | Recommend invite-only beta at launch to limit COGS exposure |
| 3 | Plugin update delivery: delta `.asar` or tied to app releases? | Delta preferred for agility; adds CDN infra requirement |

| 7 | Should there be a team / multi-seat managed tier? | Out of scope for v1; design data model to support shared log directories eventually |
